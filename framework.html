SECTION 2

Training Security Model: Two-Tier Classification

Kartel operates a two-tier training security model that provides appropriate protection based on data sensitivity. The tier is determined in consultation with the client during onboarding and formalized in the engagement contract.

STANDARD Standard Secure Training
When Used
Default for most enterprise engagements. Client brand assets, product photography, creative materials.
Training Location
Local GPUs (primary). RunPod overflow GPUs when internal capacity is exceeded or dataset size requires additional compute.
Cloud Exposure
Source data may be uploaded to a secure S3 bucket and connected to RunPod for training. All SOC 2 compliant infrastructure. Client is notified and must approve.
Model Storage
Hugging Face private repository with 2FA and auth keys. Enterprise Hub available for SOC 2 audit trails and SSO.
Contract Clause
Option B: Secure Cloud Overflow Approved
RED CODE Air-Gapped / RED CODE Training
When Used
Client-designated sensitive data. Recommended for: pre-production vehicles, unreleased products, embargoed campaigns, trade secrets.
Training Location
Local GPUs ONLY. Air-gapped infrastructure at Beverly Hills facility. Network interfaces physically disabled. No internet connectivity.
Cloud Exposure
ZERO cloud exposure. Source data never leaves the physical facility. Data transfer occurs only via encrypted physical media under chain-of-custody protocols.
Model Storage
Local storage only. Models remain on air-gapped infrastructure. Physical media transfer only under chain-of-custody.
Contract Clause
Option A: Local-Only / Air-Gapped Training
RED CODE Classification: Data classified as RED CODE includes any material that the client designates as highly sensitive, including unreleased products, pre-production assets, embargoed campaigns, trade secrets, or any material where cloud exposure represents an unacceptable risk. RED CODE classification is determined during client onboarding and documented in the engagement contract.

4 Custom Model Training

Kartel trains custom LoRA (Low-Rank Adaptation) models on client data. These models learn stylistic and structural patterns but cannot reconstruct original source assets. The training environment is governed by the Two-Tier Classification outlined above.

Training Infrastructure

Environment Description Security
Local GPUs (Primary) On-premises GPU infrastructure at Beverly Hills office. Used for all RED CODE training and as primary for all standard training. Air-gapped option available; network interfaces physically disabled for RED CODE; no internet during sensitive training
RunPod (Overflow) External GPU rental for overflow capacity when internal GPUs are at capacity or dataset size requires additional compute. NEVER used for RED CODE data. SOC 2 Type II certified; Kartel 2FA secured account; data transferred via secure S3 bucket connection; client notified and must approve before use

How RunPod Is Used (When Approved)

  • Internal GPU capacity reached or dataset requires more compute
  • Client notified and provides written approval for cloud overflow
  • Data transferred via encrypted S3 connection to RunPod instance
  • Training executed on ephemeral instance (wiped on termination)
  • Trained model (safetensor) extracted and stored on Hugging Face
  • RunPod instance terminated; no residual data

Model Storage (Hugging Face)

Trained models produce a safetensor file, which is stored on Hugging Face:

  • Private repository with authentication keys and 2FA
  • Currently Pro plan; Enterprise Hub available for clients requiring SOC 2 audit trails and SSO
  • Organization-level access controls limit who can view or download models
  • RED CODE option: Local-only model storage, never uploaded to any cloud service
Key distinction: Trained models contain only learned parameters (stylistic patterns), not recoverable source imagery or data. The model cannot reconstruct original client assets. This is mathematically analogous to a human artist learning from reference: the knowledge is retained, not the source material.

5 Media Generation (IPP + FAL)

Content generation occurs within the Internal Production Platform (IPP), which integrates third-party AI generation APIs through a secure intermediary.

Critical Architecture Detail: Client Data Isolation in Generation

Client data does NOT pass through third-party generation APIs. The custom LoRA model trained on client data contains the client's visual DNA. During generation, Kartel builds a custom workflow where prompts call the third-party API (e.g., Runway, Kling) to generate a base result, which is then processed through the client's LoRA model locally. The prompts sent to third-party APIs do not contain client data, brand assets, or proprietary content. All client-specific intelligence lives exclusively in the custom model, which remains on Kartel's infrastructure.
Component Function Security
IPP Internal production platform where all client content creation and generations occur. Artists and generative engineers work here. 2FA Google authentication; logged and gated access; Kartel employees and approved contractors only
FAL (API Gateway) Enterprise-grade vendor providing API access to third-party generation models (Runway, Adobe Firefly, Kling, Wan, etc.). Client data is NOT sent through FAL. Only generation prompts and parameters are transmitted. Client-specific intelligence is applied via local LoRA model after API response. SOC 2 Type II compliant; private account with security authentication keys; all generations distributed directly into IPP; no client data exposure
Generation Platform (GP) AI content generation engine embedded within IPP. Executes workflows, accesses LoRA models, creates assets. Inherits IPP access controls; runs within IPP environment

Performance Insights in Generation

In addition to the core generation inputs (prompts + LoRA model), the generation step can be informed by a fifth data type: Performance Insights — aggregated creative performance data from third-party vendors (e.g. VidMob) that the brand already works with. This data informs generation parameters, enabling insight-driven content creation. See Section 7 for full detail on this data type, its lifecycle, and the feedback loop.

Generation Workflow

  1. Artist or generative engineer creates a generation brief in IPP
  2. Generation Platform loads the client's trained LoRA model
  3. Performance insights (if available) are applied as generation parameters (see Section 7)
  4. Prompts and parameters sent to third-party model via FAL API
  5. Base result returned to IPP
  6. LoRA model applied locally to base output, producing brand-accurate result
  7. Result enters review/approval workflow in IPP

Generation Metadata & Audit Trail

Every generation created in the IPP captures comprehensive metadata:

  • Timestamp, user ID, and project association
  • Prompt text and parameters used
  • Performance insight inputs applied (if any)
  • LoRA model version referenced
  • Third-party model and API endpoint used
  • Output file hash and storage location

Once deliverables are distributed by the client, performance data begins flowing back from the brand's analytics vendors into the Generate phase, creating a continuous optimization loop. See Section 7 for full detail on the performance feedback cycle.

SECTION 7

Performance Insights & Feedback Loop

Performance Insights represent a fifth data type in the Kartel pipeline — distinct from the core creative asset lifecycle (Sections 1–6). This section covers what performance insights are, how they enter the pipeline, and how they create a continuous optimization loop that makes the creative engine adaptive.

What Are Performance Insights?

Performance Insights are aggregated creative performance metrics sourced from third-party vendors that the brand already works with. The primary example is VidMob, a creative intelligence platform that provides data on how distributed media (ads, social content, video) performs across channels. This data tells the generation engine what works for a specific brand's audience, enabling customized, insight-driven content creation.

Examples of performance insight data include:

  • Creative effectiveness scores by visual style, format, or messaging approach
  • Engagement metrics by audience segment and channel
  • A/B test results on creative variations
  • Platform-specific performance benchmarks (Instagram, TikTok, YouTube, etc.)
  • Trend data on emerging visual styles or content formats

Data Classification

Attribute Performance Insights
Sensitivity Level LOW Aggregated performance metrics — not core brand assets, source imagery, or PII. Fundamentally different data class from raw training data.
Source Third-party analytics vendors the brand already works with (e.g. VidMob, Meta Ads Manager, Google Analytics). Kartel does not collect this data independently.
Contains Client Data? No core brand assets, no PII, no proprietary creative materials. Contains aggregated metrics about how creative content performs in market.
Retention Retained as operational data to support the ongoing feedback loop. Not subject to NIST 800-88 destruction (not source data).

How Performance Insights Enter the Pipeline

Stage What Happens Security
Initial Ingestion (New Client) When a new client's creative intelligence engine is first built, historical performance data from the brand's analytics vendor (e.g. VidMob) is ingested into the generation step. This data informs initial generation parameters — what visual styles, formats, and messaging approaches perform best for this brand. Lower sensitivity: aggregated performance metrics, not core brand assets or PII. Received via API integration or secure file transfer from the brand's vendor.
Ongoing Feedback Loop Once the creative engine is live and deliverables are being distributed, the brand's media performance data flows back from the Deliver phase to the Generate phase. This creates a continuous optimization cycle: outputs perform → performance data captured → insights fed back → next generation refined. Same low-sensitivity classification. Performance data is aggregated — it does not contain source creative assets. Feedback loop is logged and auditable.

The Feedback Loop: Deliver → Generate

The performance feedback loop is what makes the creative engine adaptive, not just generative. It works as follows:

  1. Generate: Creative content is produced using the trained LoRA model + performance insight parameters
  2. Deliver: Approved assets are sent to the client and distributed across media channels
  3. Perform: Content runs in market across platforms (Instagram, TikTok, YouTube, programmatic, etc.)
  4. Measure: Brand's analytics vendor (VidMob, etc.) captures creative performance data
  5. Feed Back: Performance data is ingested back into the Generate phase as updated insight parameters
  6. Refine: Next generation cycle uses updated insights to produce better-performing content

This cycle repeats continuously, with each round of content creating data that improves the next.

Key Principles

  • Not core client data: Performance insights are aggregated media metrics (click rates, engagement scores, creative effectiveness data) — not brand assets, source imagery, or PII. They represent a fundamentally different and lower-sensitivity data class.
  • Sourced from brand's existing vendors: This data comes from third-party analytics platforms the brand already uses (e.g. VidMob, Meta Ads Manager data). Kartel does not collect this data independently.
  • Actionable in generation: Once the brand's creative intelligence engine is built (LoRA model trained), performance insights become parameters that guide what to generate — enabling hyper-targeted content that reflects real audience behavior.
  • Creates competitive advantage: The Deliver → Generate feedback cycle is what differentiates Kartel's creative engine from static AI generation tools. Each content cycle produces data that makes the engine smarter.

Worked Example

A CPG brand works with VidMob for creative intelligence. VidMob's performance data shows that product-in-hand shots with warm lighting outperform flat-lay photography by 2.3x on Instagram for this brand's audience. This insight is fed into the generation parameters, and the creative engine prioritizes that visual style for the next campaign batch. After the campaign runs, new performance data shows that the warm-lighting approach works even better when combined with close-crop framing. This updated insight is fed back, and the next generation cycle incorporates both parameters.

Platform Security Controls (CIP & IPP)

While Kartel pursues organizational SOC 2 certification, both internal platforms maintain enterprise-grade security controls:

  • 2FA Google Authentication on all user accounts (CIP and IPP)
  • Independent login systems for clients (CIP) and internal team (IPP)
  • Full activity logging on both platforms
  • Role-based access control with project-scoped permissions
  • Encrypted data transfer for all platform communications
  • CIP notification system warning clients not to upload RED CODE data
Kartel's position: The combination of SOC 2-certified vendor infrastructure, internal security controls (2FA, access logging, data isolation, physical security), and the two-tier training model provides enterprise-grade protection for client data. Organizational SOC 2 certification is being pursued on a timeline driven by client requirements.
